# 1. 创新点思考

## 1.1 意图驱动的系统自治

若希望面向 6G 场景形成更高等级的自适应能力，创新点应从“流水线式多智能体协作”上移到“意图持续保障的自治系统”。

这一转向的核心不是长期保存原始自然语言，而是把用户输入沉淀为一个可持续维护的意图契约。该契约需要包含目标对象、优先级、时效范围、禁止项、允许牺牲项、可接受波动范围，以及能映射到网络指标或策略对象的结构化约束。后续系统执行、监控、重规划与纠偏，都围绕这个意图契约展开。

## 1.2 在线自生长 trace 数据集

如果系统具备自治闭环能力，就不再只是被动响应用户请求，而会持续经历以下过程：意图登记、策略生成、策略执行、业务监控、偏差发现、自治重入、策略修正、效果复核。这个持续循环天然会产生大量高价值 trace。

1. 通过自治闭环持续产生原始运行 trace；
2. 通过合规监控结果、执行成败、重入原因、冲突裁决和最终保障效果，为 trace 自动附加弱监督标签；
3. 通过投影、筛选、分层采样和质量评估，将原始 trace 蒸馏为可训练数据集；
4. 形成“运行 - 产数 - 训练 - 再部署 - 再运行”的自生长数据飞轮。

## 1.3 从“固定流水线协作”转向“技能化编排协作”

多智能体之间真正缺少的，不是更底层的消息传递能力，而是稳定、可复用、可组合的协作技能。若只是把多个智能体硬编码成固定顺序的流水线，那么流程一旦扩展到监控重入、跨域协商、冲突修订、异常恢复，系统复杂度会迅速上升。

更合理的创新方向，是把协作逻辑抽象成 skill：

1. 粗粒度 skill：完成一个阶段目标，例如意图澄清、策略规划、执行诊断、保障重入；
2. 协议型 skill：规定智能体如何处理上游产物、如何提出修订请求、如何消费监控告警、如何触发自治重入；
3. 编排型 skill：决定何时调用哪些子智能体、以什么顺序协作、何时中断、何时复用已有产物。

这样做的价值在于，系统不再把 agent 本身当作唯一复用单元，而是把“经过验证的处理方式”当作复用单元。这样既能保留多智能体角色分工，又能让主智能体从固定流水线控制器，升级为面向意图与状态的全局编排智能体。

## 1.5 关于“让大模型胜任该任务是否可行”的判断

可行，但必须限定职责边界。

大模型适合承担的是高语义、跨步骤、弱结构到强结构的转换任务，例如：

1. 用户意图理解与约束抽取；
2. 监控异常的因果归纳与问题解释；
3. 多域策略候选生成与修订建议；
4. 复杂上下文下的协作编排与工具调用决策；
5. 面向运维人员的可解释报告输出。

大模型不应被直接赋予无约束的底层控制权，尤其不能让其跳过规则校验、资源可行性验证和策略对象编译，直接改写关键实时控制逻辑。原因很直接：

1. 核心网控制过程受标准和一致性约束；
2. 大模型输出存在幻觉和模式漂移风险；
3. 网络控制动作需要可验证、可回滚、可审计；
4. 强实时或高风险链路不能依赖纯生成式决策。

因此，合理定位应当是：让大模型负责“语义理解、方案生成、异常归因、协作编排”，让确定性模块负责“编译、验证、裁决、执行、监控、回滚”。

# 2. 详细设计思路

## 2.1 总体架构：目标层、执行层、保障层、数据层

建议将系统抽象为四个协同层：

1. 意图目标层：维护持续生效的意图契约，而不是只保留对话文本；
2. 系统执行层：由全局编排智能体按需调用多个技能和子智能体完成策略生成与执行；
3. 保障监控层：持续观测业务状态，判断当前运行是否偏离意图契约；
4. 数据演化层：把运行全过程投影成 trace、标签和训练样本，持续反哺模型。

其中前三层构成自治闭环，第四层构成学习闭环。

## 2.2 意图目标层：把自然语言沉淀为可持续维护的意图契约

该层建议维护一个统一的 Intent Contract，至少包含以下字段：

1. 目标对象：用户、业务流、会话、切片、区域、时段；
2. 目标类型：时延、抖动、吞吐、可靠性、切换稳定性、优先级、公平性、成本；
3. 约束强度：必须满足、尽量满足、允许牺牲；
4. 生效范围：即时、阶段性、长期；
5. 冲突规则：不同意图之间的覆盖关系、优先级和失效条件；
6. 监控映射：每个语义目标对应哪些可观测指标与判定门限；
7. 退出条件：何时视为任务完成，何时自动失效。

这里有一个关键设计点：意图目标层不能只是“保存历史输入”，而必须提供生命周期规则。否则历史意图会不断叠加，最终相互冲突，反而破坏自治能力。

## 2.3 系统执行层：由全局编排智能体按状态选择 skill，而不是固定顺序推进

系统执行层建议由一个全局编排智能体统一接管入口。其职责不是亲自完成全部推理，而是结合当前意图契约、系统状态、最近一次监控结果和历史执行上下文，动态决定调用哪些 skill。

一个可行的执行模式如下：

1. 全局编排智能体先判断当前是首次规划、局部修订、冲突协商还是故障重入；
2. 若需要意图澄清，则调用意图解析类 skill，并由对应子智能体完成结构化意图抽取；
3. 若需要策略生成，则调用规划类 skill，并由规划子智能体输出候选策略对象；
4. 若需要跨域协商，则调用协议型 skill，对冲突原因、修订请求和约束回传进行标准化；
5. 若策略通过校验，则进入执行与诊断 skill；
6. 若执行失败或保障违约，则通过重入 skill 重新进入系统执行层。

这样设计后，子智能体不再是固定流水线上的“下一站”，而是被 skill 驱动的能力节点。

## 2.4 保障监控层：从执行后检查升级为持续合规监控

保障监控层是自治系统成立的关键。它需要持续监控业务状态，并判断运行状态是否仍然满足当前意图契约。

建议引入 Compliance Monitor，其职责包括：

1. 周期性采集业务遥测、策略状态和资源状态；
2. 将监控指标映射回意图契约中的约束；
3. 判断偏差是暂时波动、持续违约、资源冲突还是策略失效；
4. 生成结构化的 Autonomous Reentry Event；
5. 把重入事件送回系统执行层，触发局部重规划或全局重规划。

重入事件建议至少包含：

1. 违约对象；
2. 违约指标与偏差幅度；
3. 可能根因；
4. 受影响策略对象；
5. 需要保留的约束；
6. 本次重入建议调用的 skill 类型。

这样一来，系统的触发源就不再只有用户，而是“用户意图 + 运行状态偏差”共同驱动。

## 2.5 可配置控制边界：配置的是策略与编排，不是任意改写基础信令过程

若面向核心网落地，系统应明确把可配置边界放在以下层面：

1. 意图翻译与策略目标建模；
2. QoS / PCC 类策略候选生成与参数调整；
3. 切片选择、优先级控制、资源重分配建议；
4. 移动性相关策略建议；
5. 监控触发后的修订顺序、协同顺序与执行路径编排。

不应把系统目标定义成“让大模型自由生成并直接执行任意核心网业务流程”，因为这在标准约束和工程安全上都不成立。更准确的目标是：在标准流程边界内，让大模型参与上层自治决策。

## 2.6 数据演化层：把自治运行过程变成训练数据生产线

数据演化层不是附属记录模块，而应被设计成与自治闭环同等重要的系统组成。

建议把运行数据分成至少五类：

1. 意图 trace：从输入到结构化意图契约的转换过程；
2. 规划 trace：候选策略生成、约束引用、冲突修订过程；
3. 执行 trace：策略下发、执行结果、失败原因；
4. 保障 trace：监控指标、违约检测、重入触发过程；
5. 恢复 trace：多轮修订后的收敛路径、最终满足情况或最终失败原因。

在此基础上，进一步构建自动标签机制：

1. 结果标签：是否满足意图；
2. 质量标签：是否存在冗余调用、错误工具选择、错误修订路径；
3. 过程标签：发生了哪类冲突、哪类重入、哪类恢复；
4. 代价标签：耗时、轮次、资源扰动、影响范围；
5. 可学习片段标签：哪些步骤适合用于监督微调，哪些适合用于偏好优化。

这样，系统产生的并不只是“日志”，而是可投影为监督学习、偏好学习和评测样本的结构化数据资产。

## 2.7 技能化设计：把 agent 能力与协作协议拆开

若后续要把多个现有阶段重构为 skill，建议区分两类 skill：

1. 任务型 skill：输入一个目标，输出一个明确产物，例如结构化意图、候选策略、执行诊断；
2. 协议型 skill：规定如何处理跨智能体交互，例如如何发起修订请求、如何声明证据不足、如何消费重入事件、如何复用已有产物。

一个成熟的自治系统，应当允许以下复用方式：

1. 同一个任务型 skill 被不同编排路径复用；
2. 同一个协议型 skill 被首次规划和重入规划共同复用；
3. 主编排智能体只感知 skill 能力边界，不依赖某一个固定 agent 的硬编码顺序。

这会让系统从“agent 编排”升级为“技能编排”，明显提高可扩展性。

## 2.8 安全与可验证性：必须把大模型关进强约束执行框架

为了让大模型在核心网场景中真正可用，系统必须具备一套强约束机制：

1. 结构化输出约束：所有中间结果都必须是类型化对象，而不是自由文本；
2. 编译与校验：先把策略候选编译成可执行对象，再做资源、冲突和一致性校验；
3. 仿真或沙箱验证：对高风险动作先在离线环境评估；
4. 执行前审计：检查动作是否触碰禁止边界；
5. 执行后保障验证：确认动作是否真正改善目标；
6. 可回滚机制：若效果恶化，能够撤销或回退；
7. 全链路可追溯：从意图到动作到结果的每一步都可审计。

这里不能采用“先放行、出错再兜底”的思路，因为那会掩盖真实问题来源，也不适合高风险控制场景。正确做法是让错误尽可能在编排前、编译前和执行前暴露出来。

## 2.9 评测思路：同时评测自治能力和产数能力

最终评测不应只看单轮成功率，而应至少覆盖两组指标：

第一组是自治控制指标：

1. 意图满足率；
2. 违约发现时延；
3. 自治重入成功率；
4. 平均收敛轮次；
5. 重规划后副作用大小。

第二组是数据生产指标：

1. 单位时间新增有效 trace 数量；
2. 可投影为训练样本的比例；
3. 自动标签一致性；
4. 高价值失败样本占比；
5. 新数据带来的模型能力增益。

如果这两组指标都成立，才能说明系统不仅会“自动执行”，还会“自动学习”。

## 2.10 最终落点

综上，建议把系统目标重新定义为：

以持续意图保障为中心，构建一个由全局编排智能体驱动、由技能化子智能体执行、由合规监控持续纠偏、并能自动沉淀高质量 trace 与训练数据的闭环自治系统。

这个定义比“做一个固定流水线多智能体系统”更强，也更接近 6G 场景下对高动态、自适应、可学习网络控制体系的真实要求。
