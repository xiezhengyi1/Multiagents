# 微调数据集 V1 方案

## Summary
现在已经有能力构建一个可用的微调数据集，但适合做的是 `V1 单Agent SFT`，先覆盖 `IEA + OSA`，格式采用 `chat-messages JSONL`，主数据源采用 `artifact JSON`，必要时用数据库表做过滤和索引。

当前不建议把 `PDA/CR/AD` 作为第一版主训练目标，也不建议把“动态图状态”作为正式训练字段，因为仓库里虽然已有 `session_stage_result / episodic_experience / agent_artifact / agent_task` 等记录，且 `runtime/interfaces` 下已有大量真实 artifact，但 `network_graph_snapshot` 目前是 `0` 条，动态图样本还不成规模。也就是说，`IEA/OSA` 已经具备建集条件，`graph-aware` 微调还没有。

## 数据集结构
最终输出为两个独立数据集文件，而不是一个混杂多任务文件：

1. `datasets/iea_sft_v1.jsonl`
2. `datasets/osa_sft_v1.jsonl`

每行一条样本，统一结构如下：

```json
{
  "sample_id": "osa:session-001:artifact-123",
  "task": "osa_sft",
  "agent": "optimization_strategy",
  "messages": [
    {"role": "system", "content": "...固定 system prompt..."},
    {"role": "user", "content": "...序列化后的结构化输入..."},
    {"role": "assistant", "content": "...目标结构化输出 JSON..."}
  ],
  "metadata": {
    "session_id": "session-001",
    "snapshot_id": "snapshot-001",
    "correlation_id": "corr-xxx",
    "source_artifact_id": "artifact-123",
    "target_artifact_id": "artifact-456",
    "round_index": 1,
    "status": "succeeded",
    "data_source": "runtime_artifact",
    "has_graph_snapshot": false
  }
}
```

具体约束：

- `messages[0]`
  - 使用当前 agent 的正式 system prompt。
  - IEA 用 `IEA_SYSTEM_PROMPT`。
  - OSA 用当前 OSA 的 system prompt。
- `messages[1]`
  - 不直接塞原始 artifact envelope。
  - 只放训练所需输入语义，去掉 `artifact_id / created_at / schema_version` 这类运行时噪声。
- `messages[2]`
  - 必须是目标对象的稳定 JSON 字符串。
  - IEA 的 target 为 `OperationIntent.model_dump(mode="json")`
  - OSA 的 target 为 `PolicyPlanDraft.model_dump(mode="json")`
- `metadata`
  - 保留追溯字段，但训练时默认不拼进 prompt。
  - 后续如要做 filtering / eval / hard-negative mining，直接依赖这些字段。

## 样本构造规则
### IEA 数据集
输入配对来源：

- request: `coordinator -> intent_encoding` 的 `OperationIntentRequest`
- response: `intent_encoding -> coordinator` 的 `OperationIntent`

用户消息固定拼装为：

```text
User input:
{payload.user_input}

Conversation context:
{payload.context or "N/A"}

Use tools when entity resolution depends on live UE context or semantic knowledge.
```

assistant 目标固定为 `OperationIntent` 的完整 JSON。

过滤规则：

- 只保留 `resolution_status == "resolved"` 的成功样本作为主训练集。
- `ambiguous / unmatched` 样本单独导出为 `iea_sft_v1_reject.jsonl`，先不进主训练集。
- 若同一 `correlation_id` 有多轮重试，只保留最终成功那一轮。
- 若同一 `user_input + snapshot_id + supi + flow_id集合` 重复，做去重，优先保留目标字段更完整的样本。

### OSA 数据集
输入配对来源：

- request: `intent_encoding -> optimization_strategy` 的 `PlanningRequest`
- response: `optimization_strategy -> intent_encoding` 的 `PolicyPlanDraft`

用户消息不再伪装成自然语言，直接用结构化文本模板：

```text
PlanningRequest:
{planning_request_json}
```

assistant 目标固定为 `PolicyPlanDraft` 的完整 JSON。

过滤规则：

- 只保留存在至少一个 policy 的样本。
- 只保留其上游 IEA 已经 `resolved` 的样本。
- 若同一 `PlanningRequest` 被重复执行，只保留最终成功输出。
- 如果 `all_policies` 明显为空、仅占位、或测试桩产物过多，单独标记为低质量样本，不进主训练集。

## 数据来源与拼接逻辑
构建时以 `runtime/interfaces` 为主，数据库只做辅助校验，不做主载荷源。

拼接逻辑固定：

- 用 `artifact_type + source_agent + target_agent + correlation_id + upstream_artifact_ids` 建立 request/response 对。
- 优先按 `correlation_id` 配对。
- 若 `correlation_id` 缺失，再退到 `upstream_artifact_ids` 与 `artifact_id` 对应。
- 若两者都缺失，则该样本直接丢弃，不做猜测匹配。

质量控制固定：

- 数据集构建脚本必须输出统计摘要：
  - 原始 artifact 数
  - 成功配对数
  - unresolved 数
  - 重复样本数
  - 被丢弃样本数
- 必须同时导出：
  - 主训练集
  - 过滤掉的 reject 集
  - 汇总报告 JSON

## Test Plan
需要验证以下场景：

- 能从真实 `runtime/interfaces` 中正确配出 IEA request/response 对。
- 能从真实 `runtime/interfaces` 中正确配出 OSA request/response 对。
- unresolved IEA 样本不会进入主训练集。
- 重试/重复 artifact 不会生成重复训练样本。
- 输出 JSONL 每行都能被 `json.loads()` 成功解析。
- `messages` 中 assistant 目标能重新反序列化为对应领域对象。
- 统计报告中的样本计数与导出文件行数一致。
- 当 `network_graph_snapshot` 为空时，`metadata.has_graph_snapshot` 恒为 `false`，且不会伪造图字段。

## Assumptions
- 第一版目标不是训练“全闭环协调器”，而是训练 `IEA` 和 `OSA` 两个最稳定的生成型 agent。
- 第一版不把 `PDA/CR/AD` 纳入主训练集，因为当前它们更像执行/诊断组件，训练收益低于 `IEA/OSA`，且标签密度不足。
- 第一版不把动态图快照作为正式输入字段，因为当前没有足够的 `network_graph_snapshot` 样本。
- 第一版使用 `artifact JSON` 作为真源，数据库只承担校验、去重和统计辅助，不反向推断缺失样本。
- 第一版输出的是“可直接用于 SFT 的 chat JSONL”，不额外设计多任务混合 schema。
